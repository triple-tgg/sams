# 13. Risk & Impact (ความเสี่ยงและผลกระทบ)

การนำระบบ SAMS Module QA มาใช้งานแทนที่ระบบเอกสารแบบเดิม ย่อมมีความเสี่ยงในกระบวนการเปลี่ยนผ่าน (Transition) และการดำเนินงาน (Operation) เอกสารส่วนนี้ประเมินความเสี่ยง ผลกระทบที่อาจเกิดขึ้น และแนวทางการป้องกัน

## ตารางการประเมินและบริหารจัดการความเสี่ยง (Risk Assessment Matrix)

| ความเสี่ยง (Risk) | โอกาสเกิด (Likelihood) | ผลกระทบ (Impact) | สาเหตุและผลกระทบ | แนวทางป้องกัน / การแก้ไข (Mitigation / Fallback) |
| :--- | :---: | :---: | :--- | :--- |
| **1. Data Migration Error** (ข้อมูลตั้งต้นผิดพลาดจากการโอนย้าย) | Medium | High | **สาเหตุ:** การ Import ไฟล์ Excel เดิม (เช่น Training Record) เข้าสู่ระบบอาจมี Cell ที่จัด Format ผิด หรือข้อมูลตกหล่น<br>**ผลกระทบ:** ทำให้วันหมดอายุผิดเพี้ยนไป ส่งผลต่อการแจ้งเตือนสิทธิ์ | - สร้างระบบ *Validation-Based Upload* เพื่อให้ระบบตรวจสอบความถูกต้องของ Format วันที่ก่อนบันทึก<br>- จัดให้มีช่วง *Parallel Run* ให้ QA เทียบข้อมูลระบบใหม่กับไฟล์ Excel เดิมก่อนขึ้น Production จริง |
| **2. Server Downtime / Network Failure** (ระบบไม่สามารถเข้าถึงได้) | Low | High | **สาเหตุ:** Server มีปัญหาขัดข้อง หรือเครือข่ายภายในองค์กรล่ม<br>**ผลกระทบ:** ไม่สามารถตรวจสอบ Authorization Matrix ของช่างซ่อมบำรุงได้ อาจทำให้ไม่สามารถออก CRS (Certificate of Release to Service) ได้ทัน ส่งผลกระทบต่อตารางการบิน (Flight Schedule) | - **Fallback Procedure:** กำหนดให้มีการทำ Data Export แบบอัตโนมัติ (เช่น ระบบส่ง PDF สรุปสถานะล่าสุดประจำสัปดาห์เข้า Email) เพื่อใช้เป็นแหล่งข้อมูลอ้างอิงฉุกเฉิน (Offline mode)<br>- มีระบบ Backup Server (High Availability) |
| **3. User Adoption Resistance** (ผู้ใช้งานต่อต้านหรือไม่ถนัดระบบใหม่) | Medium | Medium | **สาเหตุ:** QA Staff คุ้นเคยกับการพิมพ์ใน Excel และมองว่าระบบ Web-based มีขั้นตอนการคลิกเยอะกว่า<br>**ผลกระทบ:** เกิดความล่าช้าในการบันทึกข้อมูล หรือมีการละเลยไม่ยอมอัปเดตข้อมูลในระบบให้เป็นปัจจุบัน | - ออกแบบ UI ให้คล้ายกับ Spreadsheet (Table-based layout) เพื่อให้เกิดความคุ้นเคย<br>- เน้นย้ำประโยชน์เรื่อง Automated Alerts เพื่อให้ User เห็นว่าช่วยลดภาระงานในระยะยาว<br>- จัดอบรม (Training) ก่อนใช้งานจริง |
| **4. Timezone Confusion** (ความสับสนเรื่องเขตเวลาในการบันทึก) | Low | Medium | **สาเหตุ:** หาก Database บันทึกเวลาผิด Timezone หรือ UI ไม่แปลงกลับเป็น Local Time<br>**ผลกระทบ:** วันหมดอายุคลาดเคลื่อนไป 1 วัน (เช่น ข้ามเที่ยงคืน) | - บังคับใช้มาตรฐาน **UTC Timestamp** ทั้งฝั่ง Backend API และ Database<br>- เพิ่ม Unit Tests ใน Utility Function ของการแปลงวันที่ (เช่น `flightDatetime.ts`) เพื่อรับประกันความถูกต้องของการแปลง Local Time |
| **5. Unauthorized Access** (การเข้าถึงข้อมูลและแก้ไขสิทธิ์โดยไม่ได้รับอนุญาต) | Low | High | **สาเหตุ:** บัญชีผู้ใช้งานถูกขโมย หรือตั้งค่า Role สิทธิ์การเข้าถึงผิดพลาด<br>**ผลกระทบ:** ข้อมูลประวัติการฝึกอบรมหรือการอนุมัติสิทธิ์ (Authorization) ถูกแก้ไขโดยผู้ที่ไม่มีสิทธิ์ ส่งผลร้ายแรงต่อข้อกำหนดด้านความปลอดภัย (Compliance Breach) | - ใช้ระบบรักษาความปลอดภัยผ่าน JWT/Cookies (httpOnly)<br>- บังคับใช้ Role-Based Access Control (RBAC) ให้มีเพียง **QA Manager / Authorized Role เท่านั้น** ที่สามารถ Edit Matrix ได้<br>- บันทึกประวัติการแก้ไขทุกครั้ง (Audit Logging) |

## สรุป (Conclusion)
ความเสี่ยงที่มีผลกระทบสูงสุดคือ **Server Downtime** ซึ่งส่งผลกระทบต่อกระบวนการตรวจปล่อยอากาศยาน (CRS) โดยตรง ดังนั้น *Manual Fallback Procedure* (การมีเอกสาร Export สำรอง) จึงเป็นมาตรการบังคับที่ต้องดำเนินการควบคู่ไปกับระบบนี้เสมอ
